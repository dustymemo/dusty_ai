Why our dashboards are JSON, not code

How Beacon Reports turned dashboards into declarative documents you can author from a prompt — and validate before they ever hit the UI.

Most BI tools make a dashboard a pile of clicks. That's fine until you want to review a change, generate one programmatically, or let a model build it for you — and suddenly there's nothing to reason about. On Beacon Reports I made the dashboard itself the artifact: a JSON document, rendered by a small registry of ~20 widgets.

The shape

Every widget is a typed object. A dashboard is just an array of them:

[
  { "type": "stat", "label": "1M-row ingest", "value": "6 s" },
  { "type": "bar",  "label": "Report load",
    "items": [ {"label":"before","value":100}, {"label":"after","value":20} ] }
]

The renderer walks the array and maps each type to a component. Add a widget type once, and every dashboard can use it. (There's a live version of exactly this on the Playground.)

Why it matters

  • Because it's data, an LLM can author a dashboard from a prompt — and we validate the JSON before it renders.
  • Every change is a reviewable diff, not an opaque UI state.
  • Non-engineers ship dashboards in minutes, no eng ticket required.
  • Values resolve against safe, parameterized SQL — no string-built queries.

The lesson

Turning a UI into a declarative document is the same trick that makes infrastructure-as-code work: once the thing is data, everything else — generation, validation, review, versioning — comes almost for free.
