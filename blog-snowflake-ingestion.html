Cutting 1M-row ingestion from 30 min to 6 s

The idempotent stage → temp → MERGE pattern behind the RDS → Snowflake migration — and why it made onboarding a one-line PR.

Moving a pharma analytics platform off RDS meant rebuilding ingestion from scratch. The headline number — 1M rows going from ~30 minutes to ~6 seconds — sounds like magic, but it came from one boring, reliable pattern.

The pattern

Land raw data in a stage, bulk-load into a temp table, then MERGE into the target:

-- 1. stage the gzipped CSV
COPY INTO tmp_load FROM @stage/file.csv.gz;
-- 2. merge into the target (idempotent)
MERGE INTO target t USING tmp_load s
  ON t.id = s.id
  WHEN MATCHED THEN UPDATE SET ...
  WHEN NOT MATCHED THEN INSERT ...;

Why so much faster

  • csv.gzip → Snowflake is a bulk load, not row-by-row inserts. That's the bulk of the 300× speedup.
  • The MERGE is idempotent — re-running the same load is a no-op, so retries are safe.
  • No manual CSV generation or legacy queries in the path.

The payoff beyond speed

Because the whole thing is idempotent and templated, onboarding a new client — its own isolated database in a multi-tenant warehouse — collapsed from hours of manual setup to a one-line PR. Speed was the headline; the real win was making the pipeline boring and safe to operate.
